# ProPresenter Feature Inventory

This inventory translates the ProPresenter research in this folder into ChurchPresenter functionality targets. It is a planning reference, not a commitment that every target ships in the first implementation pass.

The source basis is current official Renewed Vision feature pages and knowledge-base material where accessible. The table favors implementation implications over product-copy wording; it intentionally does not reproduce articles verbatim.

## Phase Vocabulary

- **Foundation**: shared contracts that should exist before parallel feature tracks build on them.
- **Core**: needed for a useful ProPresenter-like ChurchPresenter service flow.
- **Expansion**: important for parity, polish, or more advanced operator workflows after core contracts exist.
- **Future transport**: model now so the architecture can support it later, but do not let it block local Windows output.

## Cross-Cutting Contract Checklist

Implement these contracts before workstreams diverge too far:

- `OutputScreen`, `OutputEndpoint`, `ScreenMapping`, `EndpointCapability`, `PlaceholderEndpoint`, endpoint health, and screen diagnostics.
- `OutputLayerKind`, `LayerState`, `LookPreset`, `ClearCommand`, `ClearGroup`, and per-screen layer routing.
- `RenderFrame` or equivalent scene contracts for audience screens and stage screens.
- `LiveCommand`, `LiveAction`, `ActionBatch`, `ActionResult`, source metadata, authorization metadata, and a shared command executor.
- `MediaAsset`, `AssetReference`, `MediaCue`, `AudioCue`, `MediaPlaylist`, `AudioPlaylist`, `LiveVideoInput`, storage policy, missing-media state, and cleanup reference graph.
- `Library`, `Presentation`, `Slide`, `Group`, `Arrangement`, `Playlist`, `PlaylistItem`, `PlaylistTemplate`, `Theme`, and provenance metadata.
- `Timer`, `Message`, `Prop`, `Announcement`, `Mask`, `StageScreen`, `StageLayout`, `StageMessage`, and generated-token providers.
- `CaptureSession`, capture/stream profile, capture consumer state, and capture health.
- Package/export manifests that distinguish documents, media assets, portable support/interface configuration, machine-local bindings, and destructive replacement choices.

## Source Limitations

- Current official support articles are the primary source. Some topics, especially dedicated Props, Messages, Masks, calendar scheduling, and timecode articles, are thinner than the current feature surface exposed in the product pages.
- For Props and Messages, this inventory uses current Remote, Control, timer, clear-group, and Show Controls articles as the current accessible source basis.
- For Masks, this inventory uses the current output-layer article and current multi-screen feature page as the source basis.
- The official library-creation article linked from playlist material required sign-in during earlier research. Library details here come from accessible current playlist/UI/search sources and should be revisited before detailed library-creation UI work.
- Treat the inventory as a ChurchPresenter target map, then decide per implementation phase which ProPresenter behavior is actually in scope.
