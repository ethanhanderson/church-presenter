## Learned User Preferences

- When implementing work from an **attached plan**, **do not edit the plan file**; use the existing todo list (mark items in progress and complete them rather than recreating tasks).
- For major backend/output-engine refactors, keep the current operator UI visuals and interaction patterns stable unless the user explicitly asks for a UI change; explicit exceptions should be called out first.
- New Windows desktop work targets **WinUI 3** and the **Windows App SDK** (C# packaged), not WPF, Windows Forms, or new UWP.
- When asked to save git work, **run commit and push** in this repository instead of only describing steps.
- For C#/.NET, align with `.agents/skills/dotnet-best-practices/SKILL.md` when it applies; for WinUI build/test checks, follow `.cursor/rules/windows-build-verification.mdc`. For comparing or driving the WinUI UI, use the WinApp MCP and `.cursor/skills/winapp-mcp-winui/SKILL.md`.
- For WinUI controls where contrast matters (for example selected toolbar/toggle states, acrylic sidebars, transparent panels, or slide card backgrounds), prefer **ThemeResource** and system brushes over hardcoded colors; use transparency sparingly and preserve clear light/dark contrast.
- For WinUI `DropDownButton` option pickers, prefer native `MenuFlyout` / `MenuFlyoutItem` structures; when options need rich visuals such as thumbnails, use a custom item style/template instead of replacing the menu with a generic `Flyout` + `ListView`.
- For sidebar layouts that include an **AutoSuggestBox**, let the **outer sidebar container own horizontal padding**; the suggest box and its immediate wrapper should fill the available width without adding extra inner padding.
- For theme, editor, and settings surfaces, prefer clean app-native settings panels with progressive disclosure; avoid overusing description labels, complex control surfaces, or exposing internal binding/system details unless the user explicitly asks for them.
- For temporary UI redesign scaffolding or fake sample state, keep it clearly marked and remove or revert it before finalizing the work unless the user explicitly asks to keep it.

## Learned Workspace Facts

- **Primary Windows app** lives under `apps/windows/`: `ChurchPresenter` (WinUI host), `ChurchPresenter.Application` (regular .NET application layer that should stay free of WinUI/XAML references), `ChurchPresenter.Core` (portable `.cpres` I/O). The WinUI host is physically organized under feature folders such as `Views/*`, `ViewModels/*`, `Controls/*`, `Adapters/*`, `Services/Output`, and split `Hosting/*Registration.cs` files, while XAML-facing namespaces stay stable (`ChurchPresenter.Views`, `ChurchPresenter.ViewModels`, `ChurchPresenter.Controls`) for build reliability. Root shell chrome is in `MainWindow` and keeps search plus the app page `SelectorBar` in the title bar, with media/Looks and screen toggles grouped on the right. Title-bar Editor/Reflow entries are gated by the Show page's real open presentation workspace. The Show output panel persists its interface width through `WorkspaceDto`, clamps at 360-560 DIPs, defaults/resets to 460 DIPs, and supports double-click reset on the resize handle. Solution file: `apps/windows/ChurchPresenter/ChurchPresenter.slnx`.
- **Build** (x64, from repo root): `dotnet build apps/windows/ChurchPresenter/ChurchPresenter.csproj -c Debug -p:Platform=x64`. **Tests**: `dotnet test tests/ChurchPresenter.Core.Tests/ChurchPresenter.Core.Tests.csproj` and `dotnet test tests/ChurchPresenter.App.Tests/ChurchPresenter.App.Tests.csproj`; Core/Application/test projects default to `Platform=x64` to align outputs and avoid `MSB3270`. If AppX outputs are locked, close only the running `ChurchPresenter` process; direct CLI launch can hit `REGDB_E_CLASSNOTREG`, so use Visual Studio packaged deploy/F5 for manual runtime verification when needed. `dotnet format` does not load `.slnx` yet; run it per relevant `*.csproj`. Root `Directory.Build.props` turns on analyzers and code style during ordinary `dotnet build`.
- **Project organization / docs**: `shared/` holds non-runtime artifacts (contracts, design references, assets, test fixtures), not shared app UI or services unless deliberately expanded. Shared architecture docs under `docs/architecture/` are intended to stay platform-agnostic for the native desktop product model; platform-specific organization belongs under `docs/architecture/platforms/`. The ProPresenter reference entry point is `docs/reference/propresenter/features.md`, with domain detail pages under `docs/reference/propresenter/features/`. For Microsoft / .NET / WinUI docs, use the **Microsoft Learn MCP** (see `.cursor/rules/microsoft-learn-docs.mdc`); do not use Context7 for that Microsoft stack.
- **Content library lifecycle**: Library/playlist rename, delete, duplicate, import, reorder, and playlist presentation move/add flows should use the registry/manifest services (`ILibraryRegistryService`, `IPlaylistRegistryService`) and reload canonical catalog state. Library-owned presentation deletes are destructive; playlist presentation actions are reference-only removals. Playlist entries are per-occurrence references: duplicate presentation entries are allowed, and remove/reorder/preference actions should target the selected occurrence unless the operator explicitly chooses all matching instances.
- **Legacy reference** shell is **`legacy/tauri_old/`** (Tauri + Vite + React). Do **not** assume **`legacy/tauri/`** exists in the current tree (it was removed); treat `tauri_old` and `docs/migration/` as the parity reference, not a second active app root. The legacy app's content root is repo-local under `legacy/tauri_old/content`, not the live Documents / OneDrive content folder.
- **Render engine replacement** is tracked in `docs/architecture/rendering-engine-replacement.md` and `docs/reference/propresenter/presentations-and-slides.md`: slide rendering should compile host-neutral scenes in `ChurchPresenter.Application`, route them through backend live frames, and use `tests/ChurchPresenter.Rendering.Benchmarks/ChurchPresenter.Rendering.Benchmarks.csproj` for high-layer-count performance checks. Theme integration separates ordered raw slide text blocks from visual style/layout, resolves themes through the scene compiler, keeps library themes canonical, embeds applied snapshots for portability, and supports live-linked, detach, and fork flows. File-access and stale-cache handling is planned in `docs/architecture/file-access-and-cache-resilience.md`; caches should be treated as accelerators tied to content/resource stamps, not sources of truth.
- **WinUI gotchas**: `ItemsRepeater` does not populate `DataContext` on generated elements; use patterns like `Tag` with `x:Bind` instead of `element.DataContext is T` in handlers. In typed item templates, avoid plain `{Binding}` `CommandParameter` values for strongly typed `RelayCommand<T>`; WinUI can set `Command` before `CommandParameter`, so prefer item-owned parameterless commands when command assignment may call typed `CanExecute`. `IconElement` / icon markup must use valid **Symbol** values, correct **FontIcon** glyph text, or other supported forms; invalid icon text can fail at load with **XamlParseException**. `AppBarSeparator` is for `AppBar` / `CommandBar` command collections; use theme-resource `Border` dividers for title-bar groups. For packaged startup, create and activate `MainWindow` before settings/content bootstrap, and avoid constructor-time monitor enumeration or DI cycles (for example eager cache invalidator resolution in `ContentChangeBus`) before the shell is visible. For complex `ContentDialog` forms, prefer a XAML-backed dialog with companion `.xaml.cs`; configure dialogs with `ContentDialogSetup.ConfigureForPage(...)` / page `XamlRoot` before `ShowAsync`, keep layout in XAML, and avoid constraining the outer dialog `MaxWidth` when template/content sizing preserves centering.
- **Output layers / routing**: Live output uses **separate slide and media layers**; UI and clear actions refer to **slide** vs **media** (not presentation vs media), and layer operations should avoid unnecessary reloads of other layers or the preview host. Output routing is organized around **Looks/feed presets**; the default look routes **slide** and **media** layers to all created outputs. Clear buttons and custom clear groups should dispatch backend output-layer/render-engine commands (`ClearLayers` / `ClearGroup`) rather than page-local playback flags; **Clear All** is a configurable clear group, and ProPresenter-style clear scopes live with the active Look and map to backend layers. The **mask** layer is routable from Looks/screen configuration but is not an output-preview clear target; active Look routes can carry screen-specific mask IDs and slide theme variants. The Show output preview clear UI derives primary buttons from clearable layers and secondary buttons from configured clear groups, with the clear-groups flyout opened from right-click context menus and anchored to the output preview/clear-bars container; the secondary custom-groups bar collapses when no saved groups exist, uses an explicit divider above the primary bar, and the clear-button rows use scale-aware non-`CommandBar` containers so they fit at the output panel minimum width. The outer preview/clear-bars card owns rounded clipping; the preview surface and bottom primary clear bar keep square corners inside that rounded container. Clear bar buttons should stay borderless; use `Line` elements for inter-button dividers and theme-resource strokes on surrounding `Border` containers for light/dark contrast. The custom clear-groups flyout uses a padding-free presenter with padding owned by internal panels; its left group list should sit directly below the header, stay top-aligned, use native ListView-style rows with modest padding/rounded corners, transparent unselected backgrounds, normal 16px add/delete glyphs, and hover-only delete affordances. Runtime clear overrides should persist across live-session snapshot rebuilds until a new payload replaces the cleared layer. Output transport controls follow the shared **media player registration** path; presentation-only slide surfaces must not publish **empty** active player sets over the engine-driven **media layer** registration. The media transport card is separate from the output preview/clear-bars card, stays visible with a disabled empty state when no media is active, and binds slider labels/name from a single transport display snapshot.
- **Transitions / portable Show settings**: Global slide and media **duration sliders** use **100 ms** steps; slide transitions follow per-slide override, then presentation-level only when set for that presentation, then global Show toolbar controls. Media layers should use per-item transition overrides when present, otherwise the global media fallback. Transition favorites/recents and global slide/media toolbar transitions persist through `SettingsService` into portable `Configurations/Show.json`. Media seek forward/back uses the portable `mediaSeekSeconds` Show setting, defaulting to 5 seconds and clamped to 1-60 seconds from Settings.
- **Media drawer**: **Click** a media file card to send it to the **media output layer** only; **drag** onto a Show page slide to attach or create **slide-linked media actions** (click must not run slide cues). Thumbnails use **`MediaThumbnailLoader`**: raster images decode from the file when possible; videos prefer Windows scaled/shell thumbnails, then a **`MediaComposition`** frame grab if the shell returns no usable preview.
- **Show sidebar source sections**: Libraries and Playlists in the Show page sidebar should keep `NavigationViewItemHeader` section headers visible even when no data is loaded; hide the search box until searchable content exists, and collapse its container when hidden. Empty sections should use native sidebar-scale icon/text labels instead of disabled nav items or oversized empty-state cards. Sidebar library/playlist/presentation rows behave like selectable list items for normal clicks; double-click should only toggle inline rename for the clicked row. Drag/drop insertion indicators should use row-level chrome between full sidebar items, and drop-target outlines for moving presentations onto libraries/playlists should align to the full `NavigationViewItem` bounds around the name, count badge, and chevron.
- **Slide cue activation / selection / live ring**: In the Show panel, normal slide-card click takes the slide live on the **slide output layer** without selecting it; **Ctrl+click** and right-click select only, **Shift+click** range-selects, and outside clicks clear selection only. Selection chrome and live chrome are independent; the green live ring reflects the current unsuppressed `OutputLayerKind.Slide` program payload, not announcements or selection. Clearing the presentation layer or deleting the final slide should clear selection and keep Audience Output blank until another slide is taken live. Presentations may have zero slides; newly created blank slides use a transparent background with no layers/text while preserving the default text style for new text layers.
